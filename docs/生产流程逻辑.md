# 小红书自动化 - 生产流程完整逻辑

## 总体架构

```
启动 → 确保发现页 → 主循环(最多100轮) → 每轮: 扫描目标 → 进入帖子 → 浏览 → 点赞收藏 → 返回
```

所有操作基于 AirTest 框架（ADB + javacap 截图 + uiautomator），无需人工干预。

---

## 一、启动与发现页确保

### 1.1 首次启动 `_ensure_discovery_page()`

- **重启小红书 APP**（`restart_xiaohongshu()`），确保从发现页冷启动
- 等待 APP 加载，多次截图检测（最多5次，每次间隔1秒）
- 截图验证是否在发现页（OCR 识别顶部 Tab）
- 如果不在发现页，OCR 定位"发现"Tab 文字并点击

### 1.2 页面检测 `_is_on_discovery_page()`

检测原理：OCR 识别顶部 Tab 区域（屏幕 8%-22%）中的"发现"和"同城"文字，通过**文字亮度**判断哪个是高亮的选中 Tab。
- 亮度高 = 深色文字 = 高亮选中
- "发现"高亮 → 在发现页
- "同城"高亮 → 不在发现页

## 二、主循环 `_main_loop()`（最多100轮）

每轮做一轮完整的操作：找到目标 → 进入帖子 → 浏览 → 点赞收藏 → 返回。

### 2.1 每轮开始

1. **纠错检查**：调用 `_is_on_discovery_page()`，如果不在发现页则执行 `_ensure_discovery_page()`
2. **保存截图**：记录发现页截图

### 2.2 扫描目标用户

生产模式下执行**"扫描-滑动-扫描"机制**，最多尝试3次：

```
for scan_attempt in range(3):
    调用 _find_target_post() 查找目标
    if 找到目标:
        跳出循环
    调用 _random_scroll() 短距离滑动（20%屏幕高度）
    等待 1.5-2.5 秒

如果3次都没找到:
    consecutive_no_target += 1
    再额外滑动一次（给算法推荐新内容）
    continue 进入下一轮
```

**滑动距离**：从屏幕 65% 高度滑到 45% 高度（20%屏幕），时长 150-300ms，避免一次滑过太多帖子。

### 2.3 每日操作次数检查

找到目标用户后，检查该用户今日是否已达上限：

```
can_operate(nickname, daily_limit) → (True/False, reason)
```

- `daily_limit` 从 `configs/targets.xlsx` 的"每日点赞收藏上限"列读取（默认3次）
- 追踪数据存储在 `data/daily_ops.json`，跨日自动重置
- **如果达到上限**：滑动跳过该用户，继续下一轮
- **如果未达上限**：继续执行

### 2.4 进入帖子 `_enter_post(target_info)`

有3条路径：

| 路径 | 触发条件 | 行为 |
|------|----------|------|
| 测试模式 | `self.test_mode=True` | 随机坐标点击进入 |
| OCR坐标 | `target_info` 有 `x`、`y` | 直接点击 OCR 识别到的昵称位置 |
| 降级方案 | `target_info` 无坐标 | 调用 `_click_user_nickname()` 二次 OCR 扫描 |

其中 OCR 路径最精确：`_ocr_scan_for_targets()` 会将 EasyOCR 识别到的文字与 13 个目标昵称做 fuzzy 匹配（前缀匹配，≥2字符），匹配成功后返回文字的屏幕中心坐标，直接点击即可。

### 2.5 浏览帖子 `_browse_post()`

先调用 `_detect_post_type()` 自动判断帖子类型，然后分流：

**视频帖子 `_browse_video()`**：
- 拖动进度条到约 2/3 位置（模拟快进行为）
- 随机观看 3-5 秒

**图文帖子 `_browse_images()`**：
- 调用 `_count_image_pages()` 检测图片张数（通过截图特征分析）
- 如果≤1张图：不滑动，直接返回
- 如果多张图：左滑翻页直到最后一页
  - 通过**连续2次截图不再变化**来识别末页
  - 最大滑动次数 = `max(检测到的页数+5, 20)`
  - 每页停留 1-3 秒

### 2.6 点赞收藏 `_like_and_collect()`

- **随机顺序**：50% 概率"先点赞后收藏"，50% 概率"先收藏后点赞"
- 优先使用**图像模板匹配**定位按钮位置
- 模板匹配失败时降级为**固定坐标**点击：
  - 点赞：`(width*0.72, height*0.93)`
  - 收藏：`(width*0.82, height*0.93)`
- 两个操作之间有 0.5-1.5 秒的人类化间隔

### 2.7 记录操作次数

点赞收藏完成后，调用 `increment_count(nickname)` 记录该用户今日操作次数和当前时间戳，写入 `data/daily_ops.json`。

### 2.8 保存操作截图

将当前页面截图保存到操作记录目录，用于后续审计。

### 2.9 返回发现页 `_return_to_discovery()`

- 按一次返回键
- 等待 2 秒
- 调用 `_is_on_discovery_page()` 检查是否在发现页
- 如果不在发现页（可能被推荐到其他页面），执行 `_ensure_discovery_page()` 纠错

### 2.10 随机等待

返回后发现页后，随机等待 2-5 秒，然后进入下一轮。

## 三、目标匹配系统

### 3.1 扫描流程 `_find_target_post()`

```
截图 → OCR快速扫描 → 找到? → 返回目标信息
               ↓ 没找到
              LLM视觉检测 → 找到? → 返回目标信息
               ↓ 没找到
              None
```

- **OCR优先**：用 EasyOCR 本地识别，速度快（<1秒）
- **LLM兜底**：OCR 没找到时才调用 LLM 视觉检测，速度较慢（30+秒）

### 3.2 昵称模糊匹配 `_match_nickname()`

发现页瀑布流中昵称可能被截断（显示前6-10个字符），匹配策略：

1. **完全相等**：`nickname == text`
2. **OCR文本是昵称前缀**（≥2字符）：昵称被截断，如目标"小心依依"匹配到"小心依依"
3. **昵称是OCR文本前缀**（≥2字符）：OCR识别到昵称+周围文字，如"小心依依"匹配到"小心依依 |"
4. **前3个字符相同**：足够长的前缀才能确认

**过滤规则**：
- 纯数字不匹配（如"164"不会匹配"16少爷成长记"）
- 纯emoji昵称跳过（OCR无法识别emoji，如"🌛"）

### 3.3 目标用户列表

从 `configs/targets.xlsx` 读取，5列：

| 列名 | 说明 | 示例 |
|------|------|------|
| 序号 | 编号 | 1 |
| 小红书号 | 小红书ID（可为空） | |
| 昵称 | 目标昵称 | 小心依依 |
| 匹配模式 | exact/fuzzy | fuzzy |
| 每日点赞收藏上限 | 风控限制 | 3 |

当前13个目标用户，全部使用 fuzzy 模式，默认每日上限3次。

## 四、风控系统

### 4.1 每日操作次数限制

- 文件：`workflow/daily_tracker.py`
- 存储：`data/daily_ops.json`（JSON格式）
- 结构：`{"date": "2026-05-10", "users": {"小心依依": {"count": 2, "last_time": 1746864000.0}}}`
- **跨日自动重置**：每次运行时对比日期，日期变更自动清空计数
- 检查时机：找到目标用户后、进入帖子前
- 达到上限行为：滑动跳过，继续寻找其他目标

### 4.2 操作间隔控制（3分钟）

- **同一用户两次点赞收藏之间必须间隔 ≥ 3 分钟**
- 检查时机：找到目标用户后、进入帖子前
- 间隔不足行为：跳过该用户，滑动继续寻找其他目标
- 目的：模拟人类操作节奏，防止被小红书风控系统识别为机器人

### 4.3 滑动距离控制

- 滑动距离缩短为 20% 屏幕高度，避免一次滑过太多帖子
- 滑动时长 150-300ms，更自然

---

## 五、异常处理

主循环被 try-except 包裹，发生异常时：

1. 连续按 3 次返回键（尝试回到发现页）
2. 检查是否在发现页，不在则强制重启 APP
3. 重置 `consecutive_no_target` 计数
